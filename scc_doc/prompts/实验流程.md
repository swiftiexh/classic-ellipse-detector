# Codex 分阶段执行 classic-ellipse-detector 算法改进实验指南

## 1. 总体思路

不要把完整任务一次性交给 Codex 执行。

更稳妥的方式是把任务拆成三轮：

```text
第一次：Plan 模式
只让 Codex 看项目、跑 baseline、提出候选 idea。

第二次：Goal 模式
只让 Codex 实现一个最小改进 idea，并跑小规模 sanity check。

第三次：Goal 模式
如果 sanity check 通过，再让 Codex 跑完整实验并写最终报告。
```

这样可以降低以下风险：

1. Codex 一开始就乱改核心代码；
2. baseline 没跑通却直接写改进方法；
3. idea 太大，导致实验无法完成；
4. 实验结果不可复现；
5. 文档写得完整，但数据不真实；
6. 后续 git 合并困难。

# 第一轮：Plan 模式，只做第 1–3 阶段

## 2. 使用方式

在 Codex 里选择  **Plan 模式** ，然后把下面这段发进去。

---

```text
请使用 Plan 模式。

现在只完成《classic-ellipse-detector 算法改进实验任务》的第 1–3 阶段：

1. 项目侦察；

3. 提出候选 idea。

重要限制：

- 暂时不要实现改进方法；
- 暂时不要大规模修改代码；
- 如果需要写文档，只允许写入 scc_doc；
- 如果为了记录 baseline 命令需要新增少量日志文件，可以新增，但不要修改核心代码；
- 不要进入最小实现阶段；
- 不要进入完整实验阶段；
- 不要编造实验结果。

请完成以下输出：

1. 项目结构摘要；
2. baseline 算法流程；

4. 数据集结构；
5. baseline 实验结果，如果能跑通；

11. 预计是否需要修改已有文件。

请把侦察和计划写入：

E:\scc_desk\Net\classic-ellipse-detector\scc_doc\00_project_survey.md
E:\scc_desk\Net\classic-ellipse-detector\scc_doc\01_baseline_result.md
E:\scc_desk\Net\classic-ellipse-detector\scc_doc\02_idea_selection.md

完成后停止，等待我确认，不要继续实现。
```

---

## 4. 第一轮完成后，你要检查什么

Codex 第一轮完成后，不要立刻进入实现。
你需要检查它的输出是否可靠。

| 检查项         | 你要判断什么                                   |
| -------------- | ---------------------------------------------- |
| baseline 命令  | 能不能真实运行                                 |
| 数据集路径     | 有没有找对 `datasets`                        |
| 评价指标       | 有没有找到 AP / Time                           |
| 候选 idea      | 有没有太大、太虚、太难                         |
| 新增文件计划   | 是否基本都在 `scc_experiments`和 `scc_doc` |
| 修改旧文件计划 | 有没有乱改核心代码                             |
| CNN 迁移评分   | 是否有理由，不是乱打分                         |

如果它推荐的第一个 idea 很复杂，不要直接执行。
可以让它换成第二个更低成本的 idea。

# 第二轮：Goal 模式，实现最小版本

## 5. 使用方式

确认第一轮方案没问题后，再切到  **Goal 模式** 。

这一轮只做一件事：

> 实现排名第一的改进 idea，并跑小规模 sanity check。

不要直接跑完整实验。

---

## 6. 第二轮提示词

```text
请使用 Goal 模式。

我确认你上一轮的计划。现在只实现排名第一的改进 idea。

要求：

1. 只实现最小可行版本；
2. 优先新增文件；
3. 尽量不要修改已有核心文件；
4. 如果必须修改已有文件，先说明原因，并保持最小修改；
5. 使用 wrapper 或参数开关控制新模块；
6. 不要同时实现多个 idea；
7. 不要无限调参；
8. 先做 5–20 张图像的小规模 sanity check；
9. sanity check 失败就停止，不要进入完整实验；
10. 不编造结果。

请完成：

1. 新增实验代码；
2. 跑小规模 sanity check；
3. 保存可视化结果；
4. 写入：

E:\scc_desk\Net\classic-ellipse-detector\scc_doc\03_sanity_check.md

完成后汇报：

- 新增文件；
- 修改文件；
- 运行命令；
- sanity check 是否通过；
- 结果路径；
- 是否建议进入完整实验。
```

---

## 7. 第二轮完成后，你要检查什么

第二轮完成后，重点看：

| 检查项                    | 你要判断什么               |
| ------------------------- | -------------------------- |
| 是否只实现了一个 idea     | 有没有同时做多个方向       |
| 是否主要新增文件          | 有没有大量改旧代码         |
| sanity check 是否真实运行 | 有没有命令、日志、输出路径 |
| 输出格式                  | 能不能被评价脚本读取       |
| 可视化结果                | 图片是否真的生成           |
| 耗时                      | 有没有明显变慢很多         |
| 失败处理                  | 如果失败，有没有如实记录   |

如果 sanity check 失败，不要进入完整实验。
先让 Codex 分析失败原因，必要时换第二个低成本 idea。

# 第三轮：Goal 模式，完整实验和最终报告

## 8. 使用方式

只有在 sanity check 通过后，才进入第三轮。

继续使用  **Goal 模式** 。

这一轮目标是：

> 使用完整数据集对比 baseline 和 SCC 改进方法，并写最终报告。

---

## 9. 第三轮提示词

```text
请继续使用 Goal 模式。

sanity check 已通过。现在运行完整实验。

要求：

1. baseline 和 SCC 方法使用同一批数据；
2. 使用相同评价脚本；
3. 记录 AP_0.5、AP_0.75、AP10_0.5、AP10_0.75、Time/ms；
4. 保存 baseline 和 SCC 方法的可视化对比；
5. 保存成功案例和失败案例；
6. 不编造结果；
7. 如果完整实验失败，记录报错并停止。

请写入：

E:\scc_desk\Net\classic-ellipse-detector\scc_doc\04_full_experiment.md
E:\scc_desk\Net\classic-ellipse-detector\scc_doc\05_final_report.md

最终汇报：

- baseline 结果；
- SCC 方法结果；
- 指标差值；
- 新增文件；
- 修改文件；
- 可视化路径；
- 是否建议继续这个方向；
- 是否适合迁移 CNN。
```

---

## 10. 第三轮完成后，你要检查什么

第三轮完成后，重点检查：

| 检查项                           | 你要判断什么                   |
| -------------------------------- | ------------------------------ |
| baseline 和 SCC 是否使用同一数据 | 避免不公平对比                 |
| 评价脚本是否一致                 | 避免指标不可比                 |
| 指标是否真实                     | 有没有日志或结果文件支撑       |
| Time/ms 是否合理                 | 有没有统计方式说明             |
| 可视化是否存在                   | 成功和失败案例是否都有         |
| 文档是否完整                     | 是否包含实验命令、结果表和分析 |
| 是否适合继续                     | 有没有明确下一步建议           |
| 是否适合 CNN 迁移                | 有没有具体分析，而不是空话     |

# 11. 总流程

```text
第一次：Plan 模式
只让它看项目、跑 baseline、提 idea
↓
你检查它的计划
↓
第二次：Goal 模式
只让它实现一个最小 idea，跑 sanity check
↓
你检查 sanity check
↓
第三次：Goal 模式
完整实验 + 写最终报告
```

---

# 12. 不推荐的做法

不要第一次就让 Codex 完成：

```text
读项目
→ 读论文
→ 想 idea
→ 改代码
→ 跑完整实验
→ 画图
→ 写最终报告
```

这种方式看起来省事，但风险很高：

1. 容易跑偏；
2. 容易乱改代码；
3. baseline 可能没验证；
4. 实验可能不可复现；
5. 文档可能和真实代码不一致；
6. 后续合并困难。

---

# 13. 推荐判断标准

每一轮都要先问：

```text
这一步是否真实可验证？
这一步是否会破坏已有代码？
这一步是否方便合并？
这一步是否有清楚的输出文件？
```

如果答案不清楚，就不要继续下一轮。

---

# 14. 最终建议

对 `classic-ellipse-detector` 这种研究型代码项目，推荐执行策略是：

```text
Plan 先收敛方向
Goal 再最小实现
Sanity check 再决定是否完整实验
最终报告必须基于真实结果
```

这样比一次性长任务更稳，也更适合之后迁移到 CNN。
